## 2. Goal

A polished, modern (2026-grade), single-page marketing site for Naviga Life at
`navigalife.github.io`, showing the full product range with modernized
brand-consistent imagery, downloadable per-product catalogue PDFs, compelling
condition-focused copy, testimonials, and company info — plus an intuitive
admin panel at `/admin/` (GitHub-token auth) that lets a non-technical owner
CRUD products/testimonials, reorder them, edit company info, switch between 4
design themes, and publish — with every future asset (image, catalogue)
automatically consistent with the launch brand. Nothing about the site may
suggest its content originated on IndiaMart.

## 4. Architecture (decided — do not re-litigate)

| Decision | Choice | Why |
|---|---|---|
| Rendering | Static HTML generated by `src/build.js` (Node, template literals) into `dist/` | Full SEO (content in initial HTML), zero framework risk across multi-session runs, trivially debuggable |
| Hosting | GitHub Pages, **Source: GitHub Actions**, deploy on push to `main` | Free, fits ~200 visits/mo; Actions source frees us from branch/folder constraints |
| Data / CMS persistence | JSON files in `data/` — the repo IS the database | 20 products, 20–30 edits/mo; git gives history, rollback, and zero infra |
| Admin | Static SPA at `/admin/`, auth = fine-grained GitHub PAT (this repo only, Contents:RW + Actions:RW), stored in sessionStorage (optional "remember on this device" → localStorage) | One admin user; OAuth needs a server (Decap/Sveltia proxies) — PAT keeps the whole system serverless |
| Publish flow | Admin batches all edits → ONE commit via Git Data API (blobs → tree → commit → update `main` ref) → Actions builds + deploys; admin polls run status and shows it | Atomic edits, no half-published states; "Rebuild site" button = `workflow_dispatch` |
| Catalogue PDFs | **Never image-generated.** `src/templates/catalogue.html` (A4 print template) rendered by Puppeteer in CI for every product on every build | This is the determinism answer: a new product added via admin gets a pixel-consistent catalogue automatically, forever |
| Product images | Image-gen **edit mode with the scraped photo as reference input** + a locked style-suffix prompt (§9); prompt/model recorded per asset in a manifest | Product integrity preserved (it's the real device), style locked by the constant suffix |
| Responsive images | `sharp` in build: 480/960/1440w WebP + fallback, `<picture>` + `srcset`, lazy-load below fold | Perf on mobile data |
| Fonts | Self-hosted WOFF2: **Fraunces** (display) + **Instrument Sans** (body/UI) | Distinctive editorial + banned-slop-font-free; self-hosting = no third-party request, no layout shift surprises |
| Ordering | Array order in JSON = display order; admin up/down buttons splice the array | Simplest thing that works |
| Themes | 4 presets in `data/themes.json` as CSS custom-property sets (each with light+dark variants); `data/site-config.json` picks one | Theme switch = data change = rebuild; site stays coherent because components only consume tokens |

## 6. Data schemas

`data/products.json` — array; array order = display order:

```json
{
  "id": "venogain-scd-500",
  "name": "VenoGain SCD 500",
  "tagline": "Sequential compression for DVT prevention and lymphedema care",
  "category": "DVT Pumps",
  "model": "SCD 500",
  "description": "2–4 sentence rewritten marketing description.",
  "conditions": ["DVT prevention", "Lymphedema"],
  "images": ["assets/products/venogain-scd-500/01.png"],
  "specs": [
    { "key": "Chambers", "value": "4-chamber sequential", "visible": true },
    { "key": "Pressure range", "value": "20–120 mmHg", "visible": true }
  ],
  "catalogue": true,
  "featured": false,
  "visible": true
}
```

Required: `id` (slug, immutable), `name`, `category`, `description`, ≥1 image.
Everything else optional. `specs` is the ordered custom key/value list with
per-row `visible`. `catalogue: true` → CI generates `dist/catalogues/<id>.pdf`.

`data/testimonials.json` — array; order = display order:

```json
{
  "id": "t-001",
  "type": "quote",                    // "quote" | "photo"
  "name": "Rajesh Malhotra",
  "location": "Gurugram, Haryana",
  "image": "assets/testimonials/t-001.png",
  "quote": "One or two specific sentences.",
  "context": "Post-surgical DVT prevention, 8 weeks",
  "placeholder": true
}
```

No mandatory validation beyond `id` + at least one of `quote`/`image`.

`data/company.json`: name, legalName, tagline, about (2–3 paragraphs),
phone, whatsapp, email, address, mapsUrl, hours, social links, founded,
gstDisplay (bool). All editable in admin.

`data/site-config.json`: `{ "theme": "meridian", "heroHeadline": "...", "heroSub": "...", "seo": { "title": "...", "description": "..." } }`

`data/themes.json`: 4 themes — each `{ id, label, light: {tokens}, dark: {tokens} }`
where tokens are the CSS custom properties in §8.1. Themes: `meridian`
(default, deep teal + warm paper — exact values in §8.1), `clinic` (slate
blue + cool paper), `sage` (moss green + sand), `graphite` (near-mono ink +
restrained amber accent). Derive the other three from meridian's structure;
every theme must pass the same contrast bar (§8.1).

## 7. Phases

Sequencing note: Phase 1 (scrape) is strictly first and must complete —
everything else works from its cache. If a 429 interrupts it, do Phase 2's
brand-doc work and Phase 3 scaffolding from partial cache, and resume the
scrape ≥24h later.

### Phase 1 — Scrape & archive (session 1, first ~hour)

Write `tools/scrape.py` and archive the IndiaMart presence exhaustively into
`scrape/` (gitignored).

**429-avoidance protocol (binding):**

- **Cache-first, forever**: before any request, check `scrape/manifest.json`;
  a cached URL is NEVER re-fetched. Save raw bytes immediately on receipt.
- **Mobile first**: scrape `m.indiamart.com/navigalifeventuresllp-newdelhi/`
  and its subpages (products/pview pages, about, contact) — server-rendered.
  Fall back to `www.` only for pages with no `m.` equivalent.
- **Pace**: single-threaded; random 12–20s delay between requests to any
  `*.indiamart.com` host. Total page budget is small (~15–30 pages) — the
  whole crawl fits in ~10 minutes of wall time. Do not parallelize.
- **Headers**: full realistic Chrome-on-mac set (UA, `Accept`,
  `Accept-Language: en-US,en;q=0.9`, `Referer` = the page that linked it) and
  handle brotli/gzip (`curl --compressed` semantics).
- **Images**: `*.imimg.com` is a separate host — 2–4s delays are fine. For
  each image, rewrite the size segment to `/1000x1000/` (also try original
  `/`-less variant); keep the largest that returns 200. Cache-first applies.
- **429 tripwire**: on the FIRST 429/403/captcha page from any indiamart
  host: stop all requests to indiamart hosts, record
  `{"blocked_at": iso8601, "host": ...}` in the manifest, and continue the
  session with cached material only. Resume ≥24h later.
- **Extraction**: parse from the raw cache (never re-fetch to re-parse) into
  `scrape/extracted.json`: every product (name, description text, spec
  key/values as shown, price if shown, all image URLs + local cached paths),
  company info (about text, address, contact, GST, years, certifications),
  any reviews/ratings/testimonial material, category structure.

**Checkpoint 1**: `scrape/extracted.json` complete; manifest lists every URL
with status; PROGRESS.md updated with product count + any gaps.
**Verify**: `python3 -c "import json;d=json.load(open('scrape/extracted.json'));print(len(d['products']),'products')"` — expect ≥9;
`git status --ignored -- scrape/` shows scrape/ ignored, nothing staged.

## 8. Design system (binding)

### 8.1 Tokens — theme `meridian` (default)

```css
:root { /* light */
  --bg: #F6F4EF;          /* warm paper — page background */
  --surface: #FDFCF9;     /* cards */
  --surface-2: #EEEAE2;   /* wells, table stripes, image frames */
  --ink: #182A2E;         /* headings, body at 90% */
  --ink-muted: #55666B;
  --line: #DDD6CA;        /* hairline borders */
  --primary: #0E6B63;     /* deep teal — CTAs, links, active states */
  --primary-ink: #F6FBFA; /* text on primary */
  --primary-soft: #E3EFED;/* tints, chips, hover washes */
  --accent: #B25A38;      /* terracotta — SPARINGLY: chips, small highlights, never large fills */
  --focus: #0E6B63;
}
[data-theme="dark"] {
  --bg: #10181A; --surface: #172124; --surface-2: #1E2A2E;
  --ink: #E9E6DF; --ink-muted: #9AA8AB; --line: #2A373B;
  --primary: #4FB3A6; --primary-ink: #0C1517; --primary-soft: #16302D;
  --accent: #D98E68; --focus: #4FB3A6;
}
```

Rules: components consume ONLY tokens (this is what makes the 4 themes and
dark mode coherent by construction). Body text ≥ 4.5:1 contrast, large
text/UI ≥ 3:1 — verify in both modes, all themes. Dark theme: reduce image
frame brightness with a subtle `--surface-2` wash, never invert photos.
Theme toggle: sun/moon inline SVG in nav, persists to localStorage, respects
`prefers-color-scheme` on first visit, no flash (inline head script).

### 8.2 Type & space

- Display: **Fraunces** (600/564 optical sizes for h1/h2) — hero, section
  headings, product names in overlay. Body/UI: **Instrument Sans** (400/500/600).
- Scale (desktop → mobile): h1 clamp(2.4rem→3.8rem), h2 clamp(1.8rem→2.4rem),
  h3 1.25rem, body 1.0625rem/1.65, small .875rem. Max text measure 68ch.
- Spacing on an 8px grid; section vertical rhythm `clamp(4rem, 9vw, 7.5rem)`;
  card padding 24/32px; border-radius: 10px cards, 8px controls, 999px chips.
- Hero: asymmetric editorial layout (not centered-text-over-gradient):
  headline left ~55%, supporting device imagery right, trust strip beneath.

### 8.3 Motion

- Scroll reveals via one IntersectionObserver: `opacity 0→1` +
  `translateY(14px→0)`, 320ms `cubic-bezier(0.22,1,0.36,1)`, 60ms stagger
  within a group, each fires once. Hover: card lift 2px + shadow ease 180ms.
  Overlay: 240ms scale(.985→1)+fade. Smooth scroll for anchor nav with
  `scroll-margin-top`. ALL of it gated behind `prefers-reduced-motion`.
- Nothing else moves. No parallax, no marquees, no auto-carousels.

### 8.4 Ban list (fails review)

Purple/indigo gradients or any hero gradient wash · glassmorphism ·
emoji as icons (use inline SVG, 1.5px stroke, consistent set) ·
Inter/Roboto/Arial/Space Grotesk · centered generic SaaS hero ·
auto-playing carousels · stock-photo doctors · lorem ipsum ·
skeuomorphic badges ("100% certified" seals) · more than one accent color
per surface.

## 9. Brand-consistency protocol for imagery (the determinism answer)

The owner's core worry: assets added later via admin must match the
front-loaded ones. Three mechanisms, layered:

**9.1 Locked prompt templates** (`docs/brand/IMAGE-STYLE.md`). Every image is
generated as: *[product-specific clause] + [CONSTANT style suffix]*. Author
the suffix once in Phase 2 and freeze it. Required ingredients: studio
product photography, soft diffused lighting from upper left, seamless warm
off-white background (#F6F4EF), device on a low matte plinth, slight 3/4
hero angle, shallow depth of field, no text, no people, no logos, muted
clinical palette echoing deep teal accents, 3:2 aspect. For existing
products: **edit/reference mode with the scraped photo as input** ("keep this
exact device, its shape, ports, tubing and colorway") so product integrity
survives restyling. Portrait template for photo testimonials: natural window
light, warm neutral backdrop, chest-up, gentle genuine expression, Indian
adult (vary age/gender), 1:1. Every generation appended to
`assets/products/manifest.json` with full prompt + model + date — the replay
recipe for asset #21.

**9.2 Deterministic catalogues.** PDFs are never generated by image models —
they're the Puppeteer-rendered HTML template (§Phase 5.2). New product →
identical catalogue design, automatically, in CI.

**9.3 CSS normalization frame.** Product images always render inside a fixed
3:2 frame: `background: var(--surface-2)`, `object-fit: contain`, uniform
inner padding (8%), consistent radius. Even a slightly-off future image
reads as on-brand because the frame, not the photo, carries the consistency.

**9.4 Admin embeds the protocol.** The add-product form shows the exact
prompt template (product name interpolated, copy button) — the owner never
has to remember the recipe. IMAGE-STYLE.md is the canonical reference.
